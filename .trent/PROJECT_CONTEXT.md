# Project Context: trent Agent Memory System

## Mission
Build a persistent, cross-IDE AI agent memory system baked into trent. Every developer who installs trent gets automatic memory capture, semantic search over past work, and session-start context briefing — across Cursor, Claude Code, Gemini/Antigravity, and VS Code — all stored in the existing PostgreSQL/pgvector infrastructure.

## Current Phase
**Phase 0: Research & Foundation**

Establishing platform-specific session storage locations, designing the project/user identity system, and writing the PostgreSQL schema before any code is written.

## Success Criteria

### Primary Objectives
- [ ] All 4 platforms can capture conversation turns into PostgreSQL
- [ ] Semantic search works across all platforms and projects
- [ ] memory_context() generates useful session-start briefings
- [ ] Works on Windows (primary), macOS, Linux
- [ ] Project identity survives renames/moves via UUID in .trent/
- [ ] trent_install deploys memory system to new projects automatically

### Quality Standards
- Memory ingestion never blocks any IDE (fire-and-forget)
- All LLM summarization uses local infrastructure (no third-party relay)
- Platform adapters follow registry pattern (easy to add new platforms)
- project_id + user_id as compound identity (not path-based)

### User Experience Goals
- Zero-configuration: install trent → get memory for free
- Cross-IDE: work in Cursor morning, Claude Code afternoon → unified history
- Transparent: memory_context injects relevant past context at session start

## Architecture Overview

```
Platform Capture Layer (platform-specific)
├── cursor_adapter.py     → reads state.vscdb (cursorDiskKV table)
├── claude_adapter.py     → reads ~/.claude/projects/*.jsonl
├── gemini_adapter.py     → TBD (research Phase 0)
└── vscode_adapter.py     → TBD (research Phase 0)
         ↓
Common MCP Pipeline (plugins/agent_memory.py)
├── Parse turns into user/assistant pairs
├── LLM summarize → title + description
├── Embed → pgvector (text-embedding-3-small)
└── Store in PostgreSQL (keyed by project_id + user_id)
         ↓
Shared Query Tools
├── memory_ingest_session()  — capture and store
├── memory_search()          — semantic search
├── memory_sessions()        — list recent sessions
└── memory_context()         — session-start briefing
```

## Identity System

### Project Identity
- **`project_id`**: UUID generated by `trent_install`, stored in `.trent/PROJECT_CONTEXT.md`
- Survives: project renames, moves, git clones to different paths
- Never changes after initial generation

### User Identity  
- **`user_id`**: UUID stored in `~/.trent/user_config.json` (global, per-machine)
- **`machine_id`**: Windows MachineGuid (from registry) or generated UUID
- Created on first `trent_install` run on the machine

## Scope Boundaries

### In Scope
- Cursor (Windows primary): state.vscdb reader, PS1 hooks
- Claude Code (cross-platform): JSONL reader, shell hooks
- Gemini/Antigravity (TBD after research)
- VS Code + Claude extension (TBD after research)
- PostgreSQL storage with pgvector embeddings
- MCP tools: ingest, search, sessions, context
- trent_install deployment of memory system
- AGENTS.md / CLAUDE.md template updates

### Out of Scope
- Third-party cloud relay for LLM summarization (use local/own API keys only)
- tmux dependency (Windows incompatible)
- Login/account system
- Windsurf, Zed, other IDEs (Phase 0+ only)
- Web sharing of sessions (future)

### Approved Complexity
- Python sqlite3 for reading IDE databases (stdlib, no extra deps)
- PostgreSQL + pgvector (already running in docker)
- Platform registry pattern for adapters
- Fire-and-forget HTTP for hook→MCP communication

## Key Constraints

1. **Never block any IDE**: hooks must exit in < 500ms regardless of MCP state
2. **Privacy first**: turn content never leaves local infrastructure
3. **Identity stability**: project identity cannot be path-dependent
4. **Multi-platform parity**: same memory accessible from all IDEs
